== Project Vision
Our vision is to create the "VS Code for Chinese language learning"—an extensible, powerful, and open platform that becomes the de-facto standard for digital sinology. We envision a future where anyone, regardless of their background, can achieve fluency through a combination of structured learning, personalized practice, and AI-driven feedback. {project-name} aims to be a living project, constantly evolving with the latest research in linguistics, pedagogy, and artificial intelligence.

== Problem Statement
Learning Chinese presents a unique set of challenges:

*   **The Character Barrier:** Memorizing thousands of complex characters (Hanzi).
*   **Tonal Language:** Mastering the four primary tones (and the neutral tone) which can completely change a word's meaning.
*   **Complex Grammar:** Understanding a grammatical system that differs significantly from Indo-European languages.
*   **Scarcity of Integrated Tools:** Learners often juggle multiple apps—a dictionary, a flashcard app, a character writing app, a textbook—leading to a fragmented experience.
*   **Lack of Context:** Dictionaries often provide translations without the necessary context, nuance, or example sentences.

{project-name} is architected from the ground up to address these problems directly.

=== Frontend Architecture
The frontend is built with https://astro.build[Astro], a modern static site builder that allows us to ship zero JavaScript by default. For interactive components, or "islands of interactivity," we use https://react.dev[React] (specifically, `.tsx` components). This "Islands Architecture" provides the best of both worlds: fast-loading static content and rich, dynamic user interfaces where needed.

See <<folder-structure>> for more details on the frontend codebase layout.

==== Key Libraries
*   **Astro:** Core framework for static site generation.
*   **React:** For interactive UI components (`.tsx`).
*   **Tailwind CSS:** For styling.
*   **TanStack Query:** For data fetching, caching, and state management.
*   **Zustand:** For lightweight global state management.

=== Backend Architecture
The backend consists of a set of serverless functions acting as microservices. This architecture allows for independent scaling of services based on demand. For example, the OCR engine might require more resources than the dictionary engine.

=== Database Architecture
We employ a polyglot persistence strategy, using different databases for different needs:

.Primary Database (Turso/libSQL):: Based on SQLite, it provides a fast and reliable relational store for user data, notebooks, and other structured information. See `src/lib/db.ts` for the schema and client.
.Vector Database (e.g., Pinecone):: Used by the OCR and advanced search features to find visually or semantically similar characters.
.Static JSON Chunks:: The core dictionary data is pre-compiled and split into smaller JSON files located in `public/data/chunks/`. This allows the frontend to fetch only the data it needs on demand, reducing initial load times. This is a key performance optimization.
.Cache (Redis):: A Redis layer caches frequent API responses and dictionary lookups to reduce database load and improve response times.

=== Search Engine Architecture
The search engine is a custom-built service optimized for Chinese language queries.

*   **Indexing:** All dictionary entries are indexed. See <<indexing>>.
*   **Query Parsing:** The engine can parse queries in Pinyin (with or without tones), Hanzi, and English.
*   **Fuzzy Matching:** It uses algorithms like Levenshtein distance to handle typos.
*   **Component-based Search:** Allows searching for characters by their constituent radicals or components.

// src/lib/searchService.ts
export class SearchService {
  // ...
  async search(query: string, options: SearchOptions): Promise<SearchResult[]> { // <1>
    // Step 1: Normalize and parse the query
    const parsedQuery = this.parseQuery(query);

    // Step 2: Fetch initial candidates from static chunks or cache
    let candidates = await this.fetchFromChunks(parsedQuery.initialChar);

    // Step 3: Rank results based on relevance
    const rankedResults = this.rank(candidates, parsedQuery);

    // Step 4: Return top results
    return rankedResults.slice(0, options.limit || 50);
  }
  // ...
}
----

=== OCR Architecture
The OCR engine uses a deep learning model to recognize Chinese characters in images. The process is as follows:

.   Image pre-processing (grayscale, binarization, noise reduction).
.   Contour detection to identify potential character bounding boxes.
.   Each bounding box is passed to a Convolutional Neural Network (CNN) for classification.
.   The classified characters are then looked up in our dictionary.

=== Stroke Order Engine
The stroke order data is stored as a series of SVG paths or coordinates in the static JSON files. The frontend `StrokeOrder.astro` component reads this data and uses a JavaScript animation library to render the character being drawn stroke by stroke.

=== Dictionary Engine
This is the core service that provides detailed information about a word or character. It aggregates data from our primary database and the static JSON chunks.

== Dataset Pipeline
[[dataset-pipeline]]
The quality of our dictionary is paramount. The data pipeline is a multi-stage process that ensures accuracy and richness.

=== Data Sources
Our dataset is a careful aggregation of several open-source and public domain resources:

*   **CC-CEDICT:** A public domain Chinese-to-English dictionary.
*   **Unihan Database:** Provides detailed information about Han characters, including radical-stroke counts and variants.
*   **Wikitionary:** For example sentences and etymological information.
*   **Hand-curated content:** Our team of linguists adds and refines entries.

=== Data Cleaning
The raw data from these sources is often inconsistent. The cleaning script (`scripts/import-dictionary.ts`) performs several tasks:

*   Removes duplicates.
*   Corrects formatting errors.
*   Merges entries for the same word from different sources.

=== Normalization
All data is normalized into a consistent schema. For example, all Pinyin is converted to use numbered tones (e.g., "ma1", "ma2"). See <<appendix-data-schema>> for the full schema.

=== Indexing
After cleaning and normalization, the data is indexed for the search engine. The main dictionary is split into chunks based on the starting letter of the Pinyin transcription (`a.json`, `b.json`, etc.). This is a crucial optimization for the frontend. The `scripts/split-dictionary.ts` script handles this process.

[TIP]
The entire dataset pipeline can be run locally. See the <<developer-documentation>> section for instructions.

== Caching
[[caching]]
We use a multi-level caching strategy to ensure fast response times.

*   **CDN Caching:** Static assets, including the JSON data chunks, are cached at the edge by our hosting provider (e.g., Vercel or Cloudflare).
*   **API Caching:** The API Gateway caches responses for common queries.
*   **Client-side Caching:** The frontend uses `localStorage` and `sessionStorage` to cache dictionary entries and user preferences. See `src/lib/storage.ts`.

== Performance Optimization
Performance is a core feature.

*   **Static-First:** Using Astro to deliver pre-rendered HTML.
*   **Code Splitting:** JavaScript is only loaded for the interactive components that need it.
*   **Data Sharding:** The dictionary is split into small, manageable chunks.
*   **Optimized Images:** Using modern image formats like WebP.
*   **Database Query Optimization:** All database queries are analyzed and optimized.

== Research Notes
Developing {project-name} involved significant research into computational linguistics and machine learning.

.Challenges in Pinyin-to-Hanzi Conversion footnote:[Pinyin is a romanization system, meaning one Pinyin syllable can correspond to many different Hanzi characters.]
One of the hardest problems is disambiguating Pinyin input. For example, "yishi" could mean 意识 (yìshí, consciousness), 仪式 (yíshì, ceremony), 医师 (yīshī, doctor), and many others. We use a combination of n-gram models and context from surrounding words to predict the most likely character sequence. The formula for our ranking score is roughly:
latexmath:[\\text{Score}(H|P) = \\alpha \\cdot \\log(P(H)) + (1-\\alpha) \\cdot \\log(P(H|C))]
Where latexmath:[H] is the Hanzi sequence, latexmath:[P] is the Pinyin input, and latexmath:[C] is the context.

.Semantic Search with Word Embeddings
For English-to-Chinese search, simple keyword matching is insufficient. We use pre-trained word embeddings (like BERT) to represent both the English query and the Chinese definitions in a high-dimensional vector space. A search then becomes a k-nearest neighbor problem in this space, allowing us to find semantically related terms even if they don't share any keywords.

== Known Limitations
*   The OCR engine struggles with highly stylized or calligraphic fonts.
*   The `hanviet` and `vietnamese` fields in the dictionary are community-contributed and may not be as complete or accurate as the Chinese-English definitions.
*   The Spaced Repetition System (SRS) algorithm is basic and could be improved with more advanced models of memory.

== FAQ

How is this different from Pleco or Google Translate?::
{project-name} is open-source and aims to be a platform, not just an app. Our focus is on integration and providing a holistic learning environment, from character-level details to high-level concepts, backed by a powerful and open API.

Can I use the dictionary data for my own project?::

What's the difference between "simplified" and "traditional" characters?::
Simplified Chinese characters are standardized characters prescribed in the People's Republic of China. Traditional Chinese characters are used in Taiwan, Hong Kong, and Macau. Our dictionary provides both forms.
